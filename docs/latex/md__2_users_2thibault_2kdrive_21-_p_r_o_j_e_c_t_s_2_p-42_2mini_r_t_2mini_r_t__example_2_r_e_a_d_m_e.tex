This project makes possible to create scenes such as the one below, an image of our beautiful pale blue bot called Earth.

To develop this project, we closely followed the book \href{http://raytracerchallenge.com/}{\texttt{ The Ray Tracer Challenge\+: a test-\/driven guide to your first 3D Renderer}}. The unit tests were created using \href{https://github.com/Snaipe/Criterion}{\texttt{ Criterion}}, a dead-\/simple, yet extensible, \doxylink{class_c}{C} and C++ unit testing framework. Lastly, to generate an automated documentation we used Doxygen, which is the {\itshape de facto} standard tool for generating documentation from annotated \doxylink{class_c}{C} sources and many other languages.

There are seven main folders at projects\textquotesingle{}s root directory, they are\+:


\begin{DoxyItemize}
\item {\ttfamily include}\+: contains all the necessary headers for the project
\item {\ttfamily images}\+: contains the scenes used to defend the project during the final evaluation
\item {\ttfamily lib}\+: stores our own \doxylink{class_c}{C} library with some helpful auxiliary functions, such as ft\+\_\+strlen. Thus, it contains mini\+LibX files, which is the X-\/window programming API in \doxylink{class_c}{C} used at School 42
\item {\ttfamily textures}\+: contains bump map files that can be applied to the surface of geometric shapes during renderization process
\item {\ttfamily tests}\+: contains unit tests created with \href{https://github.com/Snaipe/Criterion}{\texttt{ Criterion}} testing framework to evaluate functions in {\ttfamily src}
\item {\ttfamily documentation}\+: contains the Doxygen documentation of the project generated from header comments
\item {\ttfamily src}\+: contains the whole project implementation
\begin{DoxyItemize}
\item {\ttfamily camera}\+: contains all functions regarding camera implementation
\item {\ttfamily canvas}\+: contains all functions that works directly or indirectly with graphic data and MLX\textquotesingle{}s library manipulation
\item {\ttfamily helpers}\+: contains utils function used on the project as a whole
\item {\ttfamily lights}\+: stores all the necessary files to render a light source
\item {\ttfamily materials}\+: stores all the necessary files to render a material that has a surface color, shininess and some other characteristics
\item {\ttfamily matrices}\+: contains the files that are responsible for matrix calculations
\item {\ttfamily parser}\+: consists of functions that implement an analyzer of .rt file extension and its syntax validation
\item {\ttfamily patterns}\+: stores files that apply a striped pattern, a checkerboard pattern or a bumpmap texture on a given geometric shape
\item {\ttfamily rays}\+: stores all the necessary files for ray calculations and intersections
\item {\ttfamily shapes}\+: consists all files that stores and calculates every operation regarding the geometric shapes implemented in the project
\item {\ttfamily tuples}\+: contains the files that are responsible for vector and point calculations
\item {\ttfamily world}\+: contains everything related to the building of a ray tracing\textquotesingle{}s world. The world has objects (or shapes), point lights and an ambient color. Together, these elements makes the scene rendering possible
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md51}{}\doxysubsection{\texorpdfstring{üöÄ Optimizations}{üöÄ Optimizations}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md51}
The program underwent several optimizations to improve performance and efficiency. Two of these optimizations include\+:\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md52}{}\doxysubsubsection{\texorpdfstring{1) Memory Pool for Intersection Storage}{1) Memory Pool for Intersection Storage}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md52}
\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md53}{}\doxysubsubsubsection{\texorpdfstring{Problem}{Problem}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md53}
Previously, the linked list of ray intersections with objects was dynamically allocated using {\ttfamily malloc} and freed with {\ttfamily free} when no longer needed. This approach resulted in repeated system calls, negatively impacting the program\textquotesingle{}s performance.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md54}{}\doxysubsubsubsection{\texorpdfstring{Solution}{Solution}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md54}
To avoid the repeated calls to {\ttfamily malloc} and {\ttfamily free}, we implemented a memory pool. The memory pool consists of a pre-\/allocated block of memory used to store the nodes of the linked list. When a new list of intersections needs to be stored, the corresponding nodes are obtained from the memory pool, eliminating the need for additional memory allocation.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md55}{}\doxysubsubsubsection{\texorpdfstring{Result}{Result}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md55}
This optimization significantly reduced the time spent on memory allocation and deallocation. In a test using a standard image containing a sphere, ambient light, and a focal point light, the intersection recording was called an average of 149,000 times. Prior to the optimization, each call resulted in a {\ttfamily malloc} and {\ttfamily free} call for each node, negatively impacting the program\textquotesingle{}s performance. With the implementation of the memory pool, the {\ttfamily malloc} and {\ttfamily free} calls were eliminated, resulting in a significant performance improvement.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md56}{}\doxysubsubsection{\texorpdfstring{2) Matrix Cache for Object Transformations}{2) Matrix Cache for Object Transformations}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md56}
\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md57}{}\doxysubsubsubsection{\texorpdfstring{Problem}{Problem}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md57}
\doxylink{struct_matrix}{Matrix} operations, such as inverse and transpose calculations, accounted for a significant portion of the program\textquotesingle{}s execution time, consuming approximately 40\% of the total time. This resulted in a high performance cost, especially considering that the object transformation matrices remain constant during the rendering of static images.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md58}{}\doxysubsubsubsection{\texorpdfstring{Solution}{Solution}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md58}
To optimize performance, we implemented a matrix cache. When assigning a transformation matrix to an object, we immediately perform the necessary calculations, such as matrix inverse and transpose, and store the results in the cache. This eliminates the need to recalculate these matrices repeatedly during the rendering process.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md59}{}\doxysubsubsubsection{\texorpdfstring{Result}{Result}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md59}
With the implementation of the matrix cache, the time spent on matrix calculations was significantly reduced. The calculations are now performed only once, and the results are stored in the cache, improving the efficiency of object transformation calculations. This optimization contributed to an overall improvement in program performance, especially in scenes with multiple transformations.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md60}{}\doxysubsection{\texorpdfstring{üìñ References}{üìñ References}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2mini_r_t_2mini_r_t__example_2_r_e_a_d_m_e_autotoc_md60}

You can read the {\ttfamily io.\+Want\+Capture\+Mouse}, {\ttfamily io.\+Want\+Capture\+Keyboard} and {\ttfamily io.\+Want\+Text\+Input} flags from the \doxylink{struct_im_gui_i_o}{Im\+Gui\+IO} structure.
\begin{DoxyItemize}
\item When {\ttfamily io.\+Want\+Capture\+Mouse} is set, you need to discard/hide the mouse inputs from your underlying application.
\item When {\ttfamily io.\+Want\+Capture\+Keyboard} is set, you need to discard/hide the keyboard inputs from your underlying application.
\item When {\ttfamily io.\+Want\+Text\+Input} is set, you can notify your OS/engine to popup an on-\/screen keyboard, if available (e.\+g. on a mobile phone, or console OS).
\end{DoxyItemize}

Important\+: you should always pass your mouse/keyboard inputs to Dear Im\+Gui, regardless of the value {\ttfamily io.\+Want\+Capture\+Mouse}/{\ttfamily io.\+Want\+Capture\+Keyboard}. This is because e.\+g. we need to detect that you clicked in the void to unfocus its own windows, and other reasons.

{\bfseries{Note\+:}} The {\ttfamily io.\+Want\+Capture\+Mouse} is more correct that any manual attempt to "{}check if the mouse is hovering a window"{} (don\textquotesingle{}t do that!). It handle mouse dragging correctly (both dragging that started over your application or over a Dear Im\+Gui window) and handle e.\+g. popup and modal windows blocking inputs.

{\bfseries{Note\+:}} Those flags are updated by {\ttfamily Im\+Gui\+::\+New\+Frame()}. However it is generally more correct and easier that you poll flags from the previous frame, then submit your inputs, then call {\ttfamily New\+Frame()}. If you attempt to do the opposite (which is generally harder) you are likely going to submit your inputs after {\ttfamily New\+Frame()}, and therefore too late.

{\bfseries{Note\+:}} If you are using a touch device, you may find use for an early call to {\ttfamily Update\+Hovered\+Window\+And\+Capture\+Flags()} to correctly dispatch your initial touch. We will work on better out-\/of-\/the-\/box touch support in the future.

\begin{DoxyItemize}
\item You can share your computer mouse seamlessly with your console/tablet/phone using solutions such as \href{https://symless.com/synergy}{\texttt{ Synergy}} This is the preferred solution for developer productivity. In particular, the \href{https://github.com/symless/micro-synergy-client}{\texttt{ micro-\/synergy-\/client repository}} has simple and portable source code (u\+Synergy.\+c/.h) for a small embeddable client that you can use on any platform to connect to your host computer, based on the Synergy 1.\+x protocol. Make sure you download the Synergy 1 server on your computer. Console SDK also sometimes provide equivalent tooling or wrapper for Synergy-\/like protocols.
\item Game console users\+: consider emulating a mouse cursor with Dual\+Shock4 touch pad or a spare analog stick as a mouse-\/emulation fallback.
\item You may also use a third party solution such as \href{https://github.com/sammyfreg/netImgui}{\texttt{ net\+Imgui}}, \href{https://github.com/JordiRos/remoteimgui}{\texttt{ Remote Im\+Gui}} or \href{https://github.com/ggerganov/imgui-ws}{\texttt{ imgui-\/ws}} which sends the vertices to render over the local network, allowing you to use Dear Im\+Gui even on a screen-\/less machine. \doxylink{class_a}{A} primer on labels and the ID Stack...

Dear Im\+Gui internally need to uniquely identify UI elements. Elements that are typically not clickable (such as calls to the Text functions) don\textquotesingle{}t need an ID. Interactive widgets (such as calls to Button buttons) need a unique ID. Unique ID are used internally to track active widgets and occasionally associate state to widgets. \item You may use functions such as {\ttfamily Im\+Gui\+::\+Image()}, {\ttfamily Im\+Gui\+::\+Image\+Button()} or lower-\/level {\ttfamily Im\+Draw\+List\+::\+Add\+Image()} to emit draw calls that will use your own textures.
\item Actual textures are identified in a way that is up to the user/engine. Those identifiers are stored and passed as Im\+Texture\+ID (void\texorpdfstring{$\ast$}{*}) value.

\end{DoxyItemize}

{\bfseries{Please read documentations or tutorials on your graphics API to understand how to display textures on the screen before moving onward.}}

Long explanation\+:
\begin{DoxyItemize}
\item Dear Im\+Gui\textquotesingle{}s job is to create "{}meshes"{}, defined in a renderer-\/agnostic format made of draw commands and vertices. At the end of the frame those meshes (\doxylink{struct_im_draw_list}{Im\+Draw\+List}) will be displayed by your rendering function. They are made up of textured polygons and the code to render them is generally fairly short (a few dozen lines). In the examples/ folder we provide functions for popular graphics API (Open\+GL, DirectX, etc.).
\item Each rendering function decides on a data type to represent "{}textures"{}. The concept of what is a "{}texture"{} is entirely tied to your underlying engine/graphics API. We carry the information to identify a "{}texture"{} in the Im\+Texture\+ID type. Im\+Texture\+ID is nothing more that a void\texorpdfstring{$\ast$}{*}, aka 4/8 bytes worth of data\+: just enough to store 1 pointer or 1 integer of your choice. Dear Im\+Gui doesn\textquotesingle{}t know or understand what you are storing in Im\+Texture\+ID, it merely pass Im\+Texture\+ID values until they reach your rendering function.
\item In the \href{https://github.com/ocornut/imgui/tree/master/examples}{\texttt{ examples/}} backends, for each graphics API we decided on a type that is likely to be a good representation for specifying an image from the end-\/user perspective. This is what the {\itshape examples} rendering functions are using\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{OpenGL:}
\DoxyCodeLine{-\/\ ImTextureID\ =\ GLuint}
\DoxyCodeLine{-\/\ See\ ImGui\_ImplOpenGL3\_RenderDrawData()\ function\ in\ imgui\_impl\_opengl3.cpp}

\end{DoxyCode}
 
\begin{DoxyCode}{0}
\DoxyCodeLine{DirectX9:}
\DoxyCodeLine{-\/\ ImTextureID\ =\ LPDIRECT3DTEXTURE9}
\DoxyCodeLine{-\/\ See\ ImGui\_ImplDX9\_RenderDrawData()\ function\ in\ imgui\_impl\_dx9.cpp}

\end{DoxyCode}
 
\begin{DoxyCode}{0}
\DoxyCodeLine{DirectX11:}
\DoxyCodeLine{-\/\ ImTextureID\ =\ ID3D11ShaderResourceView*}
\DoxyCodeLine{-\/\ See\ ImGui\_ImplDX11\_RenderDrawData()\ function\ in\ imgui\_impl\_dx11.cpp}

\end{DoxyCode}
 
\begin{DoxyCode}{0}
\DoxyCodeLine{DirectX12:}

\DoxyCodeLine{-\/\ See\ ImGui\_ImplDX12\_RenderDrawData()\ function\ in\ imgui\_impl\_dx12.cpp}

 For example, in the Open\+GL example backend we store raw Open\+GL texture identifier (GLuint) inside Im\+Texture\+ID. Whereas in the Direct\+X11 example backend we store a pointer to ID3\+D11\+Shader\+Resource\+View inside Im\+Texture\+ID, which is a higher-\/level structure tying together both the texture and information about its format and how to read it.
\item If you have a custom engine built over e.\+g. Open\+GL, instead of passing GLuint around you may decide to use a high-\/level data type to carry information about the texture as well as how to display it (shaders, etc.). The decision of what to use as Im\+Texture\+ID can always be made better knowing how your codebase is designed. If your engine has high-\/level data types for "{}textures"{} and "{}material"{} then you may want to use them. If you are starting with Open\+GL or DirectX or Vulkan and haven\textquotesingle{}t built much of a rendering engine over them, keeping the default Im\+Texture\+ID representation suggested by the example backends is probably the best choice. (Advanced users may also decide to keep a low-\/level type in Im\+Texture\+ID, and use \doxylink{struct_im_draw_list}{Im\+Draw\+List} callback and pass information to their renderer)
\end{DoxyItemize}

The short answer is\+: obtain the desired DPI scale, load your fonts resized with that scale (always round down fonts size to nearest integer), and scale your Style structure accordingly using {\ttfamily style.\+Scale\+All\+Sizes()}.

Your application may want to detect DPI change and reload the fonts and reset style between frames.

Your ui code should avoid using hardcoded constants for size and positioning. Prefer to express values as multiple of reference values such as {\ttfamily Im\+Gui\+::\+Get\+Font\+Size()} or {\ttfamily Im\+Gui\+::\+Get\+Frame\+Height()}. So e.\+g. instead of seeing a hardcoded height of 500 for a given item/window, you may want to use {\ttfamily 30\texorpdfstring{$\ast$}{*}\+Im\+Gui\+::\+Get\+Font\+Size()} instead.

Down the line Dear Im\+Gui will provide a variety of standardized reference values to facilitate using this.

Applications in the {\ttfamily examples/} folder are not DPI aware partly because they are unable to load a custom font from the file-\/system (may change that in the future).

The reason DPI is not auto-\/magically solved in stock examples is that we don\textquotesingle{}t yet have a satisfying solution for the "{}multi-\/dpi"{} problem (using the {\ttfamily docking} branch\+: when multiple viewport windows are over multiple monitors using different DPI scale). The current way to handle this on the application side is\+:
\begin{DoxyItemize}
\item Create and maintain one font atlas per active DPI scale (e.\+g. by iterating {\ttfamily platform\+\_\+io.\+Monitors\mbox{[}\mbox{]}} before {\ttfamily New\+Frame()}).
\item Hook {\ttfamily platform\+\_\+io.\+On\+Changed\+Viewport()} to detect when a {\ttfamily Begin()} call makes a Dear Im\+Gui window change monitor (and therefore DPI).
\item In the hook\+: swap atlas, swap style with correctly sized one, remap the current font from one atlas to the other (may need to maintain a remapping table of your fonts at variying DPI scale).
\end{DoxyItemize}

This approach is relatively easy and functional but come with two issues\+:
\begin{DoxyItemize}
\item It\textquotesingle{}s not possibly to reliably size or position a window ahead of {\ttfamily Begin()} without knowing on which monitor it\textquotesingle{}ll land.
\item Style override may be lost during the {\ttfamily Begin()} call crossing monitor boundaries. You may need to do some custom scaling mumbo-\/jumbo if you want your {\ttfamily On\+Changed\+Viewport()} handler to preserve style overrides.
\end{DoxyItemize}

Please note that if you are not using multi-\/viewports with multi-\/monitors using different DPI scale, you can ignore all of this and use the simpler technique recommended at the top.\hypertarget{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2push__swap_2push__swap__visualizer_2bu63f6ff0ab0d9342257fc9600973b3261_autotoc_md723}{}\doxysubsubsection{\texorpdfstring{Q\+: How can I load a different font than the default?}{Q\+: How can I load a different font than the default?}}\label{md__2_users_2thibault_2kdrive_21-_p_r_o_j_e_c_t_s_2_p-42_2push__swap_2push__swap__visualizer_2bu63f6ff0ab0d9342257fc9600973b3261_autotoc_md723}
